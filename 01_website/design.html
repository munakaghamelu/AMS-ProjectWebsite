Data Storage

Currently, the dance text files are stored in the teams Azure storage account. This comes with Azures basic level of security and encryption at rest. A 30s dance will be stored as a text file ~150kB in size which means that even if we are storing the dance from 4 perspectives you will be using less than a mB of server space on the cloud. Block blob storage in Azure at the basic level of speed of access and security (which in our view is sufficient for low risk data like skeletal data) starts at £0.00061/GB per month, and so the cost of storage to the client will be minimal. One issue that would need to be fixed if the project repository was to be made public is that the authentication key for uploading and downloading to the container is available unencrypted in one of the Unity scripts. This would need to be obfuscated or removed from the project hierarchy entirely in the case that the project files were made public.

Implementation of finished functionalities

Our Unity application currently has the following features:

  • Recording a dance with 1-4 Kinects, with each Kinect connected to it's own computer
  • Uploading the dance text file from each computer to a blob storage container in our teams Azure storage account
  • Viewing the list of available files in the cloud, and downloading a selection of them to be stored locally in the project hierarchy
  • Merging and interpolating selected dance files to create a master dance text file
  • Viewing any dance text file as a skeletal representation of a dance
  • Zooming, rotating, locking on certain body parts, leaving notes, navigating through the dance with the time bar, changing the speed of playback
  • Analysing two different version of the same dance, giving the user a score of how well they performed compared to the ideal version of their dance
